1.3 业务背景

AI 大模型能力的成熟让"个性化学习路径"从概念走入产品。NotebookLM(2024)、Quizlet AI 等产品验证了"AI 处理学习资料 + 智能化输出"形态的市场可行性,资本市场对 AI+教育赛道持续看好。

同时,我们的目标用户群——22-35 岁有明确学习目标的职场人与学生——长期处于"学习焦虑"中:

  • 信息爆炸,碎片化阅读无法形成知识体系
  • ChatGPT 等通用工具能问能答,但缺乏学习追踪和闭环
  • Anki、Quizlet 等工具需要手工制作,门槛高
  • 培训机构课程标准化,无法个性化

StudyHere 切入这一市场空白:让用户上传任意资料,AI 帮其拆解、规划、闭环、追踪学习过程。

3.4 闭环概念定义

在进入"竞品功能流转分析"之前,我们需要先对"闭环"这一概念做明确定义。"闭环"是 StudyHere 与其他竞品最关键的差异化维度。

3.4.1 为什么考察"闭环"

对学习类产品而言,"用户是否能持续回来"是产品价值成立的核心。一个产品如果用户用一次就不再回来,即使单次体验再好,也无法形成商业价值。"闭环"的本质是:

  • 产品能否让用户的下一步行为自然发生
  • 产品能否把用户的输出变成下一次输入的养料
  • 产品能否形成"使用越多越好用"的飞轮

3.4.2 三种流转类型定义

类型定义用户体验后果
单向流转用户的输入经过产品处理后产生输出,输出即终点,不再产生下一步行为或新输入用户用完一次后没有继续使用的动力,易流失
半闭环产品在某一环节内部形成循环(如"卡片 → 记忆 → 卡片"),但环路狭窄,与产品的其他模块无联动用户在单一功能内可持续使用,但产品价值受限于该单一功能
全闭环用户的输出(如错题、学习行为)成为下一次循环的输入,形成"学 → 练 → 考 → 纠 → 评 → 再学"的自驱循环用户使用越多产品越懂他,持续回来动力强

四、用户研究

4.1 目标用户定位

StudyHere 服务 22-35 岁有明确学习目标(知识掌握 / 技能学习)的人群。

我们没有在 MVP 阶段对用户群做垂直收敛(如仅服务"CFA 备考者"或"程序员"),原因是:用户群收敛会限制产品定位,而用户群广带来的输入不可控风险,我们通过输入侧约束 + AI 预检 + 推荐资料兜底三层机制控制(详见第十二章)。

我们对"22-35 岁有明确学习目标"的人群做了穷举式画像分析,识别出以下主要类型。MVP 阶段我们重点支持其中三类典型用户。

画像类型占比估算MVP 支持说明

五、产品定位与策略

5.1 一句话定位

StudyHere 是帮你掌握任何知识和技能的 AI 学习闭环平台。
上传资料 + 设定目标 → AI 生成方案 → 学练考纠完整闭环 → 评估反馈 → 完成学习目标

5.2 价值主张

对谁解决什么问题核心价值
对用户把"学习"从"自我管理"变成"AI 协助"不用规划路径、不用挑题、不用怀疑自己学没学会
对市场填补 C 端学习闭环空白第一个"通用资料 + 完整学练考纠 + 评估闭环"的产品

5.3 商业模式分阶段策略

0-1 阶段不设置付费墙,原因如下:

  • MVP 验证的是"用户要不要",不是"用户付多少"
  • 早期付费墙会让 D7 / D14 留存数据失真
  • 用户尚未感受到产品价值时设置付费墙,违反 0-1 节奏
阶段商业策略目的

九、核心流程

9.1 核心设计原则

不允许任何"做完没下一步"的死路。任何最小单元(学/练/考/纠)完成后,必须有"主推荐 + 备选 1 + 备选 2"三个可选去向。

掌握度分流仅在"练"和"考"之后生效。学(闪卡)只有"懂/不懂"主观自评,无客观掌握度;纠(错题本)是输出修正,本身不产生新的掌握度数据。

9.2 智能下一步推荐矩阵

每个模块完成后的推荐逻辑,使用 MVP 阶段的规则引擎实现(不依赖 LLM,保证可解释与稳定):

当前模块完成情况主推荐备选 1备选 2
学完闪卡—练(去客观验证)看进度返回详情页
练完小测掌握度 < 60回学(薄弱知识点)错题本看进度
掌握度 60-85考(阶段测验)练更多错题本
掌握度 ≥ 85进入次日考(挑战)看进度
考完阶段测验有错题纠(错题本)回学薄弱点看进度
整体 < 60回学相关点看错题归因暂停休息
无错题且达标进入次日看进度查看完整报告
纠完错题全部清空考(再验证)进入次日看进度

9.3 当日完成判定标准

MVP 阶段采用最简明的当日完成判定:

模块是否计入"当日完成"
学(当日闪卡)✅ 必须完成
练(当日小测)✅ 必须完成
考(阶段测验)❌ 不强制每日
纠(错题本)❌ 不强制每日清空

核心判定逻辑:

  • 当日"学+练"都完成 = 当日完成
  • 当日完成 ≠ 错题清空
  • 错题清空 ≠ 当日完成
  • 错题本进度独立于当日学习计划,但错题清除会更新到当日进度展示
后续 1.0+ 阶段可考虑引入更复杂的判定逻辑(如"按错题占比"或"按阶段考试触发"),MVP 阶段先用最简版本验证核心闭环。

9.4 错题本退出与兜底机制

错题本作为容易"卡住用户"的模块,设计了完整的退出与积压兜底:

9.4.1 退出机制

  • 错题本任何时候都可以退出
  • 用户点击"返回"时弹窗确认:"还有 X 题未完成,确认退出?"
  • 退出不强制清空所有错题
  • 退出后:部分剩余的错题更新到当日进度展示,但不影响"当日完成"判定

9.4.2 积压兜底

MVP 阶段采用简单的"数量阈值"方案,不区分模块和阶段:

积压量系统行为
错题数 ≤ 30无提示,正常使用
错题数 > 30首页温和提示:"你有 X 道错题待清,要不要花 10 分钟清一下?"
错题数 > 50推荐"专项清错日",在当日学习开始前推送
30 天未清的错题自动归档(不在主列表显示,但可通过筛选查看)
MVP 阶段不引入"按知识点维度"或"按学习阶段"的复杂错题积压逻辑,保持简单。1.0+ 阶段可根据用户行为数据决定是否增加更细的分类。
